# Prototype Instructions

Run the local server yourself and open the preview in the browser available to this environment. Do not give the user server-start instructions when you can run it.

Before making substantial visual changes, use the Product Design plugin's `get-context` skill when the visual source is unclear or no longer matches the current goal. When the user gives durable prototype-specific design feedback, preferences, or decisions, record them in `AGENTS.md`.

When implementing from a selected generated mock, treat that image as the source of truth for layout, component anatomy, density, spacing, color, typography, visible content, and hierarchy.

## Codex Deck product decisions

- The prototype is a lightweight local Codex multi-account controller named Codex Deck. Windows is the currently supported platform, not part of the product positioning.
- Keep the active account pinned first; preserve addition order for all remaining accounts. Clicking another account previews it, while a separate button performs the switch.
- If Codex is running, closing it or switching accounts requires a confirmation dialog that names the running process count and warns about unsaved work.
- The collapsed form is a draggable, always-on-top-style status bar showing the active account, Codex state, and current quota.
- Quota windows are data-driven; do not hardcode a five-hour limit. V1 currently shows a seven-day quota only.
- Usage includes token totals, cache reuse, cached and uncached input, plus API-equivalent day/month cost estimates clearly labeled as estimates rather than subscription bills.
- V1 excludes task center, proxy, warm-up, automatic switching, budgets, and alerts.
- Support light/dark themes and Chinese/English. Prefer Microsoft YaHei UI for Chinese on Windows.
- Keep the collapsed status bar about 480 px wide with a neutral terminal-style Codex Deck mark that also carries the green/gray binary Codex-process dot, an 80 px ellipsized active-account name, fixed running/attention/completed icon counters, active-account quota, and a separate expand button. Brand and account areas remain draggable; do not add a second process icon.
- The title-bar minimize control collapses to the always-on-top bar. The close control opens a choice between minimizing and closing Codex Deck; closing the manager must not close Codex processes. A checked “remember my choice” stores that close-button behavior locally and skips the dialog later.
- Thread counters summarize all user-visible top-level tasks across accounts; internal subagents roll into their parent task. Running is blue, attention is amber (red when any failure exists), unread completion is green, zero is muted, and unavailable data uses a dash rather than a false zero.
- Clicking a thread-status icon opens one filtered lightweight list. The completed counter and list contain unread completions only: opening or dismissing the popover does not clear them. Clicking a completed row opens that exact local Codex task through the official `codex://threads/<thread-id>` deep link and marks only that row read; all other unread completions remain counted. A zero-count status does not open an empty popover. Clicking anywhere outside the popover closes it without changing read state. Rows show only status, one-line title, elapsed/completed time, and a non-current-account badge.
- Do not duplicate Codex sounds or Windows notifications. Completion visibility is persistent and static, without animation or flashing.
- Account aliases are optional and local. Default to the ChatGPT display name, append `· 2`, `· 3`, and so on for duplicates, and fall back to localized `Account N` only when no display name is available. Adding an account uses an isolated official-login flow and does not close or replace the currently running Codex session; switching accounts still uses the force-close safety gate.
- Only accounts explicitly saved through Codex Deck count as managed accounts. A local Codex login may identify which saved account is active, but must never be synthesized into the account list. Use the stable ChatGPT `account_id` for identity; email, display name, and plan are mutable metadata and must never merge or replace distinct accounts.
- Treat legacy accounts that lack an explicit successful-add marker as unconfirmed and hide them without deleting their stored credentials. A successful isolated official login promotes the matching legacy record or creates a new managed record; cancelled or failed login must leave the visible account list unchanged.
- Treat the successful OAuth completion and resulting isolated `auth.json` as the add-account commit point. Do not start a second isolated app-server merely to read metadata after login; decode non-sensitive name, email, plan, and account identity locally so a transient remote-service initialization error cannot turn a completed login into a visible failure.
- Collapsing the main window records its current size and position. Expanding from the status bar or tray restores that exact geometry instead of applying a default size or centering again.
- Thread counters and their filtered popovers must derive from the same top-level task collection. Completion identity includes the completion timestamp so a later completion in the same task can become unread again; keep only the latest five completions from the last 24 hours.
- Use one programmatic Tauri `start_dragging` path for the title bar and collapsed bar; do not combine it with `data-tauri-drag-region` for the same pointer event. Avoid full-window blur, large shadows, and decorative gradients that repaint during drag or resize, and skip React snapshot updates when the normalized data has not changed.
- Account day/month token totals and trends come from the account-scoped ChatGPT profile usage response. Local rollout model distribution is cross-account and must be labeled that way; do not repeat it as if it belonged to an inactive previewed account.
- API-equivalent cost estimates use the published OpenAI standard token prices and an explicitly approximate local token/model blend. They are never subscription bills, and unsupported model names must not receive invented prices.
- The main shell should grow with the native window instead of staying capped as a centered fixed-size card.
- Account profile daily totals may lag behind the current local day. Preserve the profile `stats_as_of` date, show a missing current-day bucket as pending rather than a real zero, and end the chart on the latest reported day.
- Today and month account totals use only that account's ChatGPT profile daily buckets. Local rollout history has no reliable ChatGPT account identity, so it must never be assigned to an individual account; when the official current-day bucket is missing, show Today as pending and keep local usage explicitly cross-account.
- Persist the selected interface language locally. At 960–1120 px widths, reflow cost estimates and token metrics into readable rows instead of compressing English labels into five columns.
- The selected detail-page direction is the Product Design Summary Rail: one combined four-metric usage band with the API-estimate note at its end, one dominant full-width 30-day chart, then compact local cache diagnostics and model distribution. Avoid repeating the same Today/Month metrics in separate stacked sections.
- Token charts must redraw from a `ResizeObserver` on their actual container dimensions; never rely on the next data refresh to repair a canvas stretched by native window resizing.
- Summary Rail metric cells use fixed title/value/note rows so optional note text cannot vertically shift neighboring labels or numbers. Keep the four Today/Month metrics on one baseline while the detail panel is at least 650 px wide; below that, reflow from a detail-panel container query rather than the outer window width.
- Summary Rail values must scale from the detail panel's container width and retain enough content width for large currency/token strings; values must never cross a divider at the default 1280 px window size.
- Center the four Summary Rail metric cells. Reflow them into two columns before English labels wrap or values crowd their dividers; the API-estimate explanation remains left-aligned.
- Persist the last non-compact main-window physical size and position to the local app-data window state when minimizing or quitting, and restore it on the next process launch without adding a window-state dependency or writing during live resize.
- Cache and model breakdowns come from cross-account local rollout history. Keep them visible when previewing any account, but label them as local/cross-account rather than implying that they change with the selected account.
- Favor low-repaint window surfaces: disable the native window shadow, avoid gradients and glow on persistent main-window content, cap chart raster density, and keep browser-only pointer-move handlers out of the native drag path.
- Keep native Windows 11 rounded corners through the DWM corner-preference attribute rather than transparent windows or re-enabling the native shadow.
- Keep managed credentials DPAPI-encrypted at rest. For multi-account quota refresh, decrypt only in memory and call the ChatGPT backend directly over HTTPS with the selected account's bearer token and account ID; do not write plaintext temporary auth files, switch the active Codex account, or close Codex. Cache results briefly, isolate per-account failures, and treat the internal endpoint as best-effort rather than a stable public API.
- Refresh runtime task counts through a lightweight task-only path, separate from account and usage requests. In the collapsed bar, target an update within about five seconds without increasing the frequency of network-backed quota refreshes.
- Freeze the approximate blended API pricing rate separately for each managed account and calendar month when it is first observed. Later changes to the cross-account local model/cache blend must not retroactively reprice that month's account token totals; a correction to the official token total may still change the estimate.
- Persist the selected light/dark theme as well as language. Localize the native tray menu to the saved UI language and update it immediately when the language changes.
- Because the official task list does not expose Codex UI read state, initialize existing completed tasks as a read baseline and add a completion count only after Codex Deck observes that same task transition from running/attention to completed. An idle timestamp change alone must never create a new completion.
- Keep visible product copy user-facing and concise. Do not expose implementation terms such as desktop shell, official summary API, isolated environment, or live local data when a direct status or action explains the same thing.
- The account-panel footer contains a persisted sort control instead of a passive sort description. Offer addition order, most quota remaining, and earliest reset; the active account remains pinned first under every sort mode.
- Render the account sort picker as a lightweight themed popover rather than a native select menu, so its open state matches Codex Deck on Windows. It must open upward, show the selected option, support arrow keys and Escape, and close on outside click.
- Treat the completion counter as background completions rather than official Codex unread state. When Codex is the foreground Windows app, suppress the most recently completed transition as the task the user is most likely viewing, while preserving any other simultaneous completions; when Codex is in the background, count every observed running/attention-to-completed transition.
